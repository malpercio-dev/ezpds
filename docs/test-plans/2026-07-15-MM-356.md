# Hosted Custos MCP — first slice — Test Requirements

Maps every acceptance criterion for the **first end-to-end slice** of the hosted
Custos MCP (design plan [`docs/design-plans/2026-07-14-hosted-custos-mcp.md`](../design-plans/2026-07-14-hosted-custos-mcp.md),
epic [MM-356](https://linear.app/malpercio/issue/MM-356)) to its verification
method — an **automated test** (unit | integration | e2e) at a specific location,
or **documented human verification** where automation is insufficient.

This is the **test leg** of the design → test-requirements → phases triad (per
[`docs/archive/README.md`](../archive/README.md)). The **design leg** is the plan
above; the **phases leg** is [`docs/implementation-plans/2026-07-15-MM-356/`](../implementation-plans/2026-07-15-MM-356/),
whose four phases map onto the Phase-1 children below. Each AC names the phase +
task that lands its test. Decisions this slice implements are fixed by
[ADR-0023](../architecture/decisions/0023-sovereign-child-agent-identities.md)
(sovereign child identities) and
[ADR-0024](../architecture/decisions/0024-hosted-agent-credential-forwarding.md)
(credential forwarding); several ACs below exist specifically to hold those ADRs
to their word.

## Scope of the slice

The smallest thing that proves the reframing works end to end:

1. **Mint a sovereign child agent identity** (own DID + repo + handle; recovery
   key held in the parent's Obsign wallet) as a child of a parent account on this
   PDS — [MM-368](https://linear.app/malpercio/issue/MM-368).
2. **A credential-forwarding Streamable-HTTP MCP sidecar** (`mcp.obsign.org`) that
   reuses the `tools/mcp/` tool surface, terminates only the MCP transport, and
   forwards the caller's token — [MM-369](https://linear.app/malpercio/issue/MM-369).
3. **`create_post` end to end** over that sidecar against staging, attributed to
   the child agent — [MM-370](https://linear.app/malpercio/issue/MM-370) (acceptance-defining).
4. **No regression** of the shipped self-hosted *acts-as-you* stdio MCP —
   [MM-367](https://linear.app/malpercio/issue/MM-367).

Everything past this slice (nested handles, custom domains, Iroh self-host,
in-PDS consolidation) is out of scope and tracked separately (design plan §5, §7;
epic Phase 2+).

## Conventions

This slice straddles two codebases, so it uses two test-placement conventions.

- **Rust (PDS minting, MM-368).** No separate `tests/` dir for these crates;
  tests live inline in `#[cfg(test)] mod tests` at the bottom of the source file
  they exercise (e.g. `crates/pds/src/routes/agent_identity.rs`,
  `crates/pds/src/routes/agent_child.rs` new). Route tests drive the assembled
  router via `crate::app::app(state)` + `tower::ServiceExt::oneshot`, using the
  shared helpers in `crates/pds/src/routes/test_utils.rs`
  (`state_with_master_key`, `seed_account_with_repo`, `seed_account_with_signing_key`,
  `access_jwt`, `body_json`) and `wiremock::MockServer` standing in for
  `plc.directory` (the `sovereign_session.rs` / `agent_identity.rs` test modules
  are the reference pattern).
- **Node (sidecar, MM-369 / MM-370).** Tests live under `tools/mcp-sidecar/test/`
  (new package), run by `node test/run.ts` like the existing `tools/mcp/test/`
  suite. End-to-end cases spawn a **hermetic local PDS** with the existing
  `spawnPds` / `startMockPlc` / `provisionAccount` / `confirmClaim` harness
  (`tools/mcp/test/harness.ts`) — reused, not reinvented — and drive the sidecar
  over its HTTP transport with an MCP `StreamableHTTPClientTransport`.

**Test types.**

- **unit** — a pure/near-pure function tested in-process with hand-built inputs;
  no HTTP, no network.
- **integration** — Rust: drives `app(state).oneshot(...)` with wiremock for
  `plc.directory`. Node: drives the sidecar in-process (or over loopback) with a
  fake/forwarded token, asserting forwarding behavior without a real PDS.
- **e2e / live** — a check against a **real deployed PDS** (staging,
  `ezpds-staging.up.railway.app`) or the hermetic local PDS the conformance
  harness spawns. Reserved for behavior no mock can faithfully reproduce
  (real minting → real firehose, real OAuth handshake, real attribution in the
  repo).

**Forwarding is asserted by absence as well as presence.** ADR-0024 forbids
durable secrets. Several sidecar ACs are *negative* assertions — no token on
disk, no credential in logs, nothing surviving a restart — and are as load-bearing
as the positive ones.

**Minting is gated by the ownership graph.** ADR-0023: a child can only be minted
under a parent account that exists on this PDS. The gate is not a bolted-on check;
tests assert the *absence* of a parent account makes minting impossible (design
plan §8).

## AC1: Agent-as-child-identity minting (MM-368)

The parent owner mints a sovereign child: own DID, own repo, own handle, recovery
key in the parent's wallet, day-to-day signing delegated and revocable.

| Case | Type | Test location | Description |
|------|------|---------------|-------------|
| **AC1.1** Success — parent owner mints a child; child gets its own DID + repo + handle | integration | `crates/pds/src/routes/agent_child.rs` (new) | Seed a parent account (`seed_account_with_repo`); the authenticated parent posts the mint request; assert a distinct child DID, a bootstrapped child repo (genesis commit staged to the firehose), and an allocated handle are returned — none equal to the parent's. (Phase 1 Task 3) |
| **AC1.2** Success — child genesis carries the wallet-supplied recovery/rotation key | integration/unit | `crates/pds/src/routes/agent_child.rs` + `crates/crypto/src/plc.rs` | The mint request supplies a client-signed `plcOp` (or wallet-supplied rotation key set) whose `rotationKeys` are the wallet's, not the operator's; assert `verify_genesis_op` accepts it and the persisted `did_documents` rotation set matches the wallet key — the operator key never appears as a rotation key. (Phase 1 Task 2–3) |
| **AC1.3** Success — child handle is a flat subdomain in the served namespace, reserved names refused | unit | `crates/pds/src/identity/handle.rs` | A flat `<name>.<served-domain>` handle passes `validate_handle`; a nested `a.b.<served-domain>` and a reserved name (`is_reserved_name`) are refused (nested handles are Phase 2, design plan §2). (Phase 1 Task 1) |
| **AC1.4** Failure — minting without a parent account on this PDS is refused | integration | `crates/pds/src/routes/agent_child.rs` | An unauthenticated request, and an authenticated request whose subject has no local parent account, both fail (401/403) with no child written — the ownership-graph gate (design plan §8, ADR-0023). (Phase 1 Task 3) |
| **AC1.5** Success — parent↔child ownership link persisted and enumerable | integration | `crates/pds/src/db/agent_auth.rs` (or new `agent_children` query mod) | After a mint, the child row records its parent DID; the parent can list its children and the link survives reload. (Phase 1 Task 2) |
| **AC1.6** Success — day-to-day signing is delegated, scope-clamped, revocable — not durable full authority | integration | `crates/pds/src/routes/agent_child.rs` + `crates/pds/src/db/agent_auth.rs` | The child's usable credential is a scope-clamped, expiring capability (reuses the auth.md assertion/token machinery, `RegistrationType`); assert the granted scopes are the clamped agent profile and the token is short-lived, not a standing full session. (Phase 1 Task 3) |
| **AC1.7** Success — revoking the child stops it acting without surrendering recovery | integration | `crates/pds/src/routes/agent_child.rs` + `revoke_agent_identity` | Parent revokes the child; a subsequent exchange/token mint fails `access_denied`; assert the recovery/rotation key (wallet-held) is untouched by revocation — revocation kills the delegated capability only (ADR-0023 custody ladder). (Phase 1 Task 4) |

**Live check flag:** AC1.1's "genesis commit reaches the firehose" and the DID
becoming resolvable through `plc.directory` are confirmed live (HV-1) — wiremock
proves the op is *built and staged* correctly, not that the real directory accepts
it.

## AC2: Credential-forwarding Streamable-HTTP sidecar scaffold (MM-369)

Reuse the `tools/mcp/` tool surface; swap stdio → Streamable HTTP; swap the
singleton `AgentSession` for a per-caller map; forward the caller's token and cache
nothing durable (ADR-0024).

| Case | Type | Test location | Description |
|------|------|---------------|-------------|
| **AC2.1** Success — tool surface served over Streamable HTTP, not stdio | e2e | `tools/mcp-sidecar/test/transport.test.ts` | An MCP `StreamableHTTPClientTransport` client connects to the sidecar's HTTP listener, lists tools, and sees the same tool names the stdio server exposes (`whoami`, `create_post`, `get_record`, `list_records`, `search_timeline`, `account_status`). (Phase 2 Task 3) |
| **AC2.2** Success — each caller authenticates via OAuth against Custos; no durable credential persisted | e2e | `tools/mcp-sidecar/test/forwarding.test.ts` | Drive a tool call carrying a caller-obtained bearer; after the call, assert **no** credential file exists under any state dir and **no** on-disk cache was written — the `0600` state-file path of the stdio server is *not* present in the sidecar. (Phase 2 Task 2) |
| **AC2.3** Success — per-caller state is in-memory, session-scoped, dies on restart | integration | `tools/mcp-sidecar/test/sessions.test.ts` | Establish two caller sessions; assert their state is isolated (one caller's token/DID never visible to the other) and that tearing down / restarting the sidecar process leaves no recoverable session state. (Phase 2 Task 2) |
| **AC2.4** Success — singleton `AgentSession` replaced by a per-caller map keyed by authenticated caller | unit | `tools/mcp-sidecar/test/registry.test.ts` | The session registry returns the same session object for the same authenticated caller within a session and distinct objects for distinct callers; an unauthenticated request gets none. (Phase 2 Task 1) |
| **AC2.5** Success — caller token rides each tool call to the PDS; nothing durable cached | integration | `tools/mcp-sidecar/test/forwarding.test.ts` | With a stub PDS capturing inbound requests, assert every forwarded XRPC call carries the caller's `Authorization: Bearer <token>` and that the sidecar holds no token after the request resolves (in-memory only, request-scoped). (Phase 2 Task 2) |
| **AC2.6** Success — logs/traces/errors redact authorization material | unit | `tools/mcp-sidecar/test/redaction.test.ts` | Exercise the log/error formatter with a token-bearing request/failure; assert no bearer/assertion substring appears in any emitted log line or error surfaced to the client (ADR-0024 consequence — DB avoidance alone is insufficient). (Phase 2 Task 4) |
| **AC2.7** Success — deployable as a third Railway service over private networking | inspection | `tools/mcp-sidecar/` config + `docs/deploy.md` | The sidecar reads its PDS origin from config (a `*.railway.internal` private URL in the co-located tier) and its own public origin; assert config-parse rejects an unset PDS origin and the deploy surface is documented (mirrors `sites/marketing/` third-service pattern, design plan §5). (Phase 2 Task 5) |

**Live check flag:** AC2.1/AC2.2 exercise the sidecar against the hermetic local
PDS the conformance harness spawns; the *real* MCP-spec OAuth handshake against
staging's authorization-server metadata is confirmed live (HV-3).

## AC3: `create_post` end to end over the sidecar on staging (MM-370) — acceptance-defining

| Case | Type | Test location | Description |
|------|------|---------------|-------------|
| **AC3.1** Success — `create_post` over HTTP publishes an `app.bsky.feed.post` to the child's repo | e2e | `tools/mcp-sidecar/test/create_post.test.ts` (hermetic) + HV-2 (staging) | Over the sidecar's HTTP transport, call `create_post`; assert a record is created and readable back via `get_record`/`com.atproto.repo.getRecord`. Hermetic run proves the mechanism; HV-2 proves it against real staging. (Phase 3 Task 2) |
| **AC3.2** Success — the post is attributed to the child agent DID, not the parent | e2e | `tools/mcp-sidecar/test/create_post.test.ts` + HV-2 | Assert the created record's repo/author is the **child** DID; the parent's repo is unchanged. This is the whole point of the sovereign-child model (ADR-0023). (Phase 3 Task 2) |
| **AC3.3** Failure — a forwarded token outside granted scope is refused legibly | e2e | `tools/mcp-sidecar/test/scope.test.ts` | Forward a token whose scopes exclude the write; assert the PDS returns `403 InsufficientScope` and the sidecar relays a legible, scope-naming error (parity with the stdio `relayError` behavior), not a stack trace. (Phase 3 Task 3) |
| **AC3.4** Failure — revoking the agent mid-session fails subsequent calls legibly | e2e | `tools/mcp-sidecar/test/revocation.test.ts` | Revoke the child (or its delegated capability) between two calls; assert the second `create_post` fails with a legible revoked/expired message and no partial write. **Live check flag:** AC3.1/AC3.2 are the Definition of Done and are only *fully*
demonstrated live against staging (HV-2). The hermetic e2e proves each mechanism;
the live pass proves they compose against the real deployment and firehose.

The shipped stdio MCP (`tools/mcp/`) must keep working unchanged; the acts-as-you

(design plan §1, ADR-0023).

| Case | Type | Test location | Description |
|------|------|---------------|-------------|
| **AC4.1** Success — the stdio conformance suite stays green | e2e | `tools/mcp/test/conformance.test.ts` (`just mcp-test`) | The existing hermetic conformance suite (onboard → claim confirm → `create_post` → whoami/reads) passes unchanged after this slice lands — the regression gate for the self-hosted path. (Phase 4 Task 3) |
| **AC4.2** Success — acts-as-you is documented first-class, self-host default | inspection | `tools/mcp/README.md` + `docs/…` | The docs state the attribution × hosting matrix explicitly: self-hosted acts-as-you is a supported, encouraged mode (top-left of the §1 matrix), not a fallback; hosted default is the sovereign child. Cross-links ADR-0023/0024. (Phase 4 Task 1) |
| **AC4.3** Success — no change to stdio transport, `0600` cache, or singleton session for the self-hosted path | inspection + e2e | `tools/mcp/src/*` git diff + AC4.1 | Assert `tools/mcp/`'s `StdioServerTransport`, on-disk `0600` credential cache (`state.ts`), and singleton `AgentSession` are untouched by this slice — the sidecar is a *sibling* package, not a rewrite of the stdio server. (Phase 4 Task 2) |

**Live check flag:** none — AC4 is fully covered by the existing hermetic
conformance suite plus documentation inspection. It asserts *this* codebase's
non-regression, which the harness drives deterministically.

## Human Verification

The hermetic harness fully covers this slice's own logic — minting, forwarding,
per-caller isolation, redaction, and the non-regression gate — because it
deterministically supplies whatever the PLC directory / caller "would" return. What
it **cannot** prove is that the *real* PLC directory, staging PDS, and MCP-spec
OAuth flow behave as the tests assume. The following need a one-time live pass.

### HV-1 — A minted child DID is real: genesis reaches the firehose and resolves in the PLC directory (underpins AC1.1, AC1.2)
- **Why not automated:** wiremock accepts the genesis op by fiat; it proves the op
  is *built and staged*, not that the real `plc.directory` accepts it or that the
  child becomes resolvable and its repo is federatable.
- **Approach:** Against staging, mint a child of a real staging account; confirm the
  child DID resolves via the real PLC directory, the genesis `#commit`/`#sync`/`#account`
  frames appear on the firehose, and the child repo is describable
  (`com.atproto.repo.describeRepo`). Confirm the `rotationKeys` in the resolved DID
  document are the wallet's, not the operator's.

### HV-2 — `create_post` over the hosted sidecar publishes to the child's repo on staging (Definition of Done — AC3.1, AC3.2)
- **Why not automated:** the headline guarantee — an agent, acting as its own
  sovereign identity, posts through the hosted forwarding tier — is only *fully*
  demonstrated against the real staging PDS + AppView, since attribution and
  federation are defined relative to the live network.
- **Approach:** With the sidecar deployed as the third staging Railway service
  reaching the PDS over private networking, connect an MCP client, complete the
  OAuth handshake, and call `create_post`. Confirm the post appears in the **child
  agent's** repo/feed (not the parent's) in a real Bluesky client, and that the
  audit log attributes it to the agent registration.

### HV-3 — The real MCP-spec OAuth handshake against Custos's AS metadata works (confirms AC2.1, AC2.2)
- **Why not automated:** the hermetic harness fronts the PDS with a throwaway TLS
  proxy and can stub discovery; a live check confirms the sidecar drives the *real*
  `oauth-protected-resource` → `oauth-authorization-server` walk and that a caller
  token obtained through it is accepted per-request with nothing cached.
- **Approach:** Point an MCP client at the deployed sidecar, complete the real OAuth
  authorization against staging, and confirm: (a) a tool call succeeds with the
  obtained token, (b) restarting the sidecar forces re-authorization (no durable
  session survived), and (c) no credential material appears in the sidecar's logs.

### HV-4 — Credential-forwarding negative posture holds under a real deployment (confirms AC2.2, AC2.5, AC2.6)
- **Why not automated:** "holds nothing durable" is a whole-service property; unit
  tests assert individual code paths, but only inspecting a running deployment
  proves no path writes a secret to disk, DB, or logs.
- **Approach:** On the deployed staging sidecar, exercise several tool calls, then
  inspect the container filesystem, any attached volume, and the log stream for
  bearer/assertion material; confirm none is present and a restart yields an empty
  session table.

### Note on what is *not* verified here
Nested handles (`writer.alice.obsign.org`), custom domains, the Iroh self-host
tier, and folding the endpoint into axum are **out of scope** for this slice
(design plan §2, §5, §7; epic Phase 2+). Their ACs will be authored with their own
triads when those phases are scheduled. This document covers only the smallest
end-to-end slice.
