# PDS Phase 0.1 — Reference-PDS Gap Analysis & Competitive Review

Date: 2026-06-30
Author: Phase 0.1 end-of-phase review
Method: codebase inventory (grounded in `crates/`) → multi-source web research (reference PDS + cocoon + tranquil + others) → synthesis → adversarial verification of every claimed gap against the actual code.

This document is the durable research record behind the Wave 7 issues filed during the phase-0.1 review (MM-207…MM-222). It is split into: (1) goal confirmation, (2) where we stand, (3) reference-PDS gaps → Linear issues, (4) competitive analysis with sources, (5) net-new feature candidates, (6) top-3 recommendations, (7) full source list.

## 1. Phase 0.1 goal — confirmed

Per [`docs/unified-milestone-map.md`](unified-milestone-map.md) §2.1, **v0.1 = "Mobile-Only PDS"**: the PDS is a *full* PDS — it "hosts repo, serves XRPC, **signs commits**, emits firehose." The user creates an ATProto identity from their iPhone and logs into Bluesky. The PDS **signs commits server-side** with a per-account P-256 key; the user holds the primary `did:plc` rotation key (Secure Enclave) and a 2-of-3 Shamir recovery split backs recovery.

Desktop-holds-the-signing-key sovereignty is explicitly **v2.0+** ("Pluggable signer: desktop-remote … PDS no longer signs," contingent on ATProto protocol evolution). So the original framing is correct: **v0.1 is a full operational PDS + identity wallet where the PDS signs commits; the Mac app signing its own commits is a later phase.**

## 2. Where we stand

**Done:** Waves 0–6 are complete. The implemented surface (verified in `crates/pds/src/app.rs`) — standard `com.atproto.*` methods, plus the OAuth provider, the custom `/v1/*` provisioning API, and proxied surfaces:

- **server.*:** describeServer, createSession, getSession, getServiceAuth, refreshSession, deleteSession, activateAccount, deactivateAccount, requestPasswordReset, resetPassword
- **identity.*:** resolveHandle
- **repo.*:** createRecord, putRecord, deleteRecord, applyWrites, getRecord, listRecords, describeRepo, uploadBlob (incl. swapRecord/swapCommit concurrency, historical `cid`)
- **sync.*:** getRepo (incl. incremental `since`), getRecord (proof CAR + exclusion proofs), getLatestCommit, getRepoStatus (active/deactivated/suspended/takendown derivation), listRepos, listBlobs, getBlob, subscribeRepos (durable sequencer, cursor replay, `#commit`/`#account`)
- **app.bsky.actor:** getPreferences/putPreferences stored locally
- **Proxied non-`com.atproto.*` surfaces:** `app.bsky.*` → AppView proxy; `chat.bsky.*` → chat proxy (service-auth JWT minted from the user's repo key)
- Full **OAuth provider** (PAR/authorize/token/JWKS/metadata, DPoP, ES256), plus a custom `/v1/*` mobile provisioning + device-signed `did:plc` genesis ceremony, Iroh tunnel, outbound `requestCrawl`, firehose retention sweep.

**Federation status:** ezpds **already federates** for a newly-minted identity — firehose + `getRepo` + `requestCrawl` work, and the network accepts its per-account P-256 signing keys (externally corroborated by millipds, which also defaults to P-256 and federates). So nothing strictly blocks a *new* user from participating.

**Remaining pre-existing backlog relevant to v0.1** (before this review): MM-198 (PDS self-onboard to relay on account bootstrap), MM-131 (`/v1/transfer/complete` — device-swap flow incomplete), MM-205/206 (firehose durability follow-ups), MM-197 (preferences app-password scope filtering). Wave 8 (auth.md / agentic auth, MM-166…177) is net-new agent-auth, beyond reference-PDS scope.

## 3. Reference-PDS gaps → Wave 7 issues filed

Every gap below was **confirmed against the code** (grep + router read). Severity legend: **network/sovereignty** (a normal user operation is broken or won't propagate), **migration** (blocks PDS↔PDS account migration / credible exit), **client-parity** (standard clients/tools expect it; core works without), **safety/parity** (moderation/takedown tooling expects it), **interop/nice-to-have**.

| Issue | Gap | Severity | Note |
|---|---|---|---|
| **MM-208** | `com.atproto.identity.updateHandle` | network/sovereignty | Users can't change handle; custom `/v1/handles` only rewrites the local DID doc — no PLC op, no `#identity`. |
| **MM-209** | Firehose `#identity` (and `#sync`) frames | network/sovereignty | Only `#commit`/`#account` emitted; identity changes don't propagate. Pairs with MM-208. |
| **MM-207** | **Account migration epic (PDS↔PDS)** | migration | None of the standard handshake exists; "credible exit" + "bring your identity in" both blocked. |
| MM-218 | `server.createAccount` (+ existing-DID migration mode) | migration | Also unblocks standard third-party client/CLI onboarding. |
| MM-219 | `server.reserveSigningKey` | migration | Custom `/v1/repo-signing-key` exists but is pending-session-gated + non-standard shape. |
| MM-220 | `repo.importRepo` + `repo.listMissingBlobs` | migration | Export works (`sync.getRepo`); no import path at all. |
| MM-221 | `server.checkAccountStatus` | migration | Counts already available in `db/blocks.rs`/`db/blobs.rs`. |
| MM-222 | identity PLC-signing surface (getRecommendedDidCredentials, signPlcOperation, requestPlcOperationSignature, submitPlcOperation) | migration | **Crypto already exists** (`build_did_plc_rotation_op`, `verify_plc_operation`); only the HTTP/email/submit layer is missing. | **MM-211** | Outbound email + email confirm/update | client-parity | **Password-reset email is currently stubbed** (logs token); also gates the migration PLC-signature email (MM-222) and account-delete. |

| MM-213 | `sync.getBlocks` | client-parity | Block store + CAR assembly already exist. |

| MM-217 | Invite-code XRPC + `temp.checkHandleAvailability` | interop/nice-to-have | ezpds uses its own claim-code system; only matters for standard-client signup interop. |

### Federation checklist (non-endpoint), status

1. **Handle resolution (both directions)** — ✅ `resolveHandle` (DB→DNS TXT→well-known) + `.well-known/atproto-did`. ⚠️ but handle *changes* don't propagate (MM-208/209).
2. **did:plc + rotation keys + service entry** — ✅ minted in the genesis ceremony; ⚠️ no in-band PLC update path (MM-222).
3. **Announce via requestCrawl** — ✅ outbound `crawler.rs`; ⚠️ bootstrap-on-account-create is MM-198. Early-access federation also needs hostname + contact email registered with the relay operator (manual/ops).
4. **Firehose + verifiable repo serving** — ✅ `#commit`/`#account` + sync reads; ⚠️ missing `#identity`/`#sync` (MM-209), `getBlocks` (MM-213).
5. **Contact email + outbound email** — ⚠️ describeServer advertises contact, but the outbound email path is **stubbed** (password-reset logs the token; no confirm/update endpoints) — MM-211.
6. **OAuth vs app-passwords** — ✅ full OAuth provider (forward path); ⚠️ no app passwords (MM-210, legacy clients).
7. **Labeler / moderation** — optional for a plain PDS; report-proxy + takedown enforcement are MM-215/216.

## 4. Competitive analysis (non-reference PDS implementations)

Researched implementations, with what each does **beyond** the reference PDS. Sources are inline and consolidated in §7.

### cocoon — Go, by haileyok (`github.com/haileyok/cocoon`)

Experimental but self-dogfooded (author runs her main account on it) and federates; ~48 XRPC endpoints incl. migration. Beyond-reference features:
- **Pluggable DB: SQLite (default) or PostgreSQL** (Postgres uses LISTEN/NOTIFY for the sequencer).
- **S3-compatible blob store with CDN redirect** — `getBlob` returns a 302 to the CDN instead of proxying bytes.
- **Built-in hourly SQLite→S3 backups** baked into the server.
- **Prometheus metrics** — relays connected, relay sends by event kind, repo operations by kind.
- Turnkey Docker Compose + Caddy (auto-HTTPS), auto key-gen + auto-invite; admin CLI subcommands.
- Notably **omits app passwords** (OAuth-first) — same stance ezpds currently has.

### tranquil — Rust, single static binary + Postgres (`tranquil.farm`, mirror `github.com/Bogay/tranquil-pds`)

Community project positioned as a **feature superset** of the reference PDS. Beyond-reference:
- **Passkeys + 2FA** (WebAuthn/FIDO2, TOTP, backup codes, trusted devices).
- **Granular OAuth scopes with a consent UI** (uncheck individual scopes).
- **Scoped app passwords** (read-only / post-only / custom — same permission model as OAuth).
- **Account delegation** — others manage an account at configurable permission levels, with audit logging.
- **Multi-channel notifications** (email, Discord, Telegram, Signal).
- **did:web** (hosted subdomain or BYO domain) + built-in web UI (account mgmt, repo browser, admin).

### rsky-pds — Rust, Blacksky (`github.com/blacksky-algorithms/rsky`)

Most mature non-reference Rust PDS; runs in production behind managed hosts. Postgres + S3 blob storage (chosen explicitly for cloud portability), Mailgun email, whole-stack Rust (own relay/firehose/crypto/repo crates). Supports migration + PLC signing. A coherent reference architecture for a server-side-signing Rust PDS at scale.

### millipds — Python, by David Buchanan (`github.com/DavidBuchanan314/millipds`)

WIP but federates. **Defaults to NIST P-256 signing keys** (validates ezpds's core bet that P-256 federates), serves blobs with **HTTP range requests + Content-Disposition**, supports `swapRecord` in `applyWrites` and base64 DAG-CBOR record values, and targets minimal hardware (Raspberry Pi, UNIX-domain-socket deploy).

### cirrus — TypeScript on Cloudflare Workers (`github.com/ascorbic/cirrus`)

Single-user, edge-native (Durable Objects + R2). **Passwordless passkey auth**, OAuth 2.1 (PKCE/DPoP/PAR), and — most relevant — **explicit signing-key custody UX with did-method-specific recovery messaging** (did:web loses crypto continuity on key loss; did:plc recovers via a registered rotation key). A model for honest custody/recovery UX.

### bluepds — Rust (Axum + Atrium) (`github.com/DrChat/bluepds`)

Early-stage; built on the **Atrium Rust ATProto SDK** (lexicon reuse path), pluggable storage backend trait (file → Azure Blob/B2 planned), telemetry exporters, PLC-signing endpoints + firehose producer.

### Managed hosts (Blacksky, Northsky, EuroSky)

Live PDS hosting. Ship **first-class guided migration tooling as a product** (move off Bluesky's PDS without losing followers/posts/identity), and plan **custom private-data storage** + community-governed moderation layers.

## 5. Net-new feature candidates (report only — not filed as gaps)

These are *beyond* the reference PDS — ideas worth considering for ezpds, mapped to the identity-wallet (Obsign) brief. Sources in §7.

1. **Passkey / Secure-Enclave-anchored session auth** (tranquil, cirrus) — ezpds is already device-key-centric in the *provisioning ceremony*, but the standard `createSession` path is still a single argon2id password. Promote the device/biometric to a first-class session factor.
2. **Granular, scoped credentials** (tranquil) — OAuth scopes with a consent UI + scoped app passwords. This is the principled way to add the missing app-password surface (MM-210) and fits Obsign's "see and control exactly what touches your identity."
3. **Account delegation / guardian model** (tranquil) — configurable-permission management of an identity with audit log; synergistic with ezpds's existing Shamir recovery-share custody.
4. **S3/CDN blob offload with `getBlob` 302-redirect** (cocoon, rsky) — keeps the small Railway PDS off the media bandwidth path.
5. **HTTP range requests + Content-Disposition on `getBlob`** (millipds) — cheap mobile streaming/scrub win.
6. **Prometheus metrics for federation health** (cocoon) — relay connectivity + commit throughput; ezpds ships only `/xrpc/_health`.
7. **Multi-channel security alerts** (tranquil) — for "a PLC op was requested" / "a new device paired," don't depend on a single email channel.
8. **Guided in-wallet migration** (Blacksky/Northsky) — once MM-207 lands, wrap it in an Obsign onboarding flow with honest custody/recovery messaging.
9. **did:web option** (tranquil, cocoon) — domain-anchored identity for the sovereignty-minded, with the honest caveat (cirrus) that did:web loses crypto continuity on key loss.
10. **Storage-backend trait** (bluepds, rsky) — clean per-account SQLite + object-store split as ezpds grows past one VM.

## 6. Top 3 recommendations

### #1 — Close the handle-change loop: `updateHandle` + `#identity` firehose (MM-208 + MM-209)

The single most-visible broken *normal-user* operation. Changing a handle is table-stakes sovereignty, the Bluesky app exposes it directly, and ezpds can't do it network-visibly today. It's also small and self-contained: the crypto (`build_did_plc_rotation_op`) exists, and adding the `#identity` frame is a clean, well-scoped firehose change. Highest value-to-effort ratio for "fully participate as a normal user."

### #2 — Make account migration real, designed around client-side rotation keys (MM-207 epic)

"Credible exit" and "bring your existing identity in" are core to ATProto's value proposition and to a *sovereign identity wallet's* pitch specifically — a wallet users can't leave (or arrive at) undercuts the entire thesis. ezpds already has the export legs and the PLC crypto; the work is the import/PLC-signing/status HTTP layer. Crucially, design `signPlcOperation`/migration around ezpds's **client-side rotation-key custody** (the wallet already calls these NSIDs on peer PDSes), which is a *differentiator* vs. reference PDSes that hold the rotation key outright. Sequence behind #1 since they share the PLC-signing surface.

### #3 — Fix the email path, then add scoped credentials (MM-211 → MM-210)

First fix the **stubbed outbound email** (MM-211) — it silently breaks password reset *today* and gates both migration (PLC-signature email) and account deletion. Then add **scoped app passwords / OAuth-scope consent** (MM-210, informed by tranquil): it restores legacy-client compatibility *and* doubles as a flagship Obsign feature ("issue this device a post-only credential; see and revoke exactly what each app can do"). This turns a parity gap into a product differentiator.

> Net-new features from §5 (passkey sessions, delegation, CDN offload, metrics, multi-channel alerts) are recorded for the roadmap but intentionally **not** filed as gaps — they're enhancements beyond reference parity, best revisited after the network-participation and migration gaps close.

## 8. Protocol evolution watch

*Added 2026-07-04. §§1–7 above are the 2026-06-30 point-in-time review; much of the gap table has since landed (updateHandle, `#identity`/`#sync` + `prevData`, PLC-signing surface, importRepo, getBlocks, app passwords, deleteAccount, oauth-protected-resource, takedown writer, rate limiting — see Linear for live status). This section tracks the upstream protocol changes that affect ezpds's roadmap, to be re-checked periodically.*

### 8.1 Keyless PDS / user-owned signing keys — the v2.0 sovereignty path is forming upstream

- **[Discussion #4739](https://github.com/bluesky-social/atproto/discussions/4739)** (julienrbrt, Mar 2026): add an optional **`#atproto_service`** verification method so a DID doc can carry a *service-auth JWT signing key* (PDS-held) separate from the *commit signing key* `#atproto` (which can then be **user-held, off-server**). Motivation is explicitly **keyless/"BYOK" PDSes** — without it, every background proxied request from a keyless PDS needs a user-presence gesture. Working implementation in **[indigo PR #1356](https://github.com/bluesky-social/indigo/pull/1356)** + an example keyless PDS. Status: proposal with momentum, no formal acceptance, no objections recorded.
- **Substrate already landed:** [did-method-plc PR #101](https://github.com/did-method-plc/did-method-plc/pull/101) relaxed verification-method key formats and set a **10-key limit** per PLC DID doc — the identity layer can hold multiple keys today.
- **What this means for ezpds:** the *remote-signer* architecture (wallet device holds the sole `#atproto` commit key; PDS builds unsigned commits, ships the hash to the device for Secure-Enclave P-256 signing, and signs its own service-auth with `#atproto_service`) is becoming a *supported network pattern*, not a workaround. This de-risks v2.0 "signing sovereignty" substantially. Writes still gate on device availability — that's irreducible.
- **Still genuinely gated on protocol evolution:** *multi-device* signing where each device holds its own independent commit key the network verifies. `#atproto_service` separates key *purposes*; relays still verify every commit against the single `#atproto` key. No current proposal changes the consuming side. Interoperable interim options: copy the one key across devices (loses per-device revocation) or threshold/MPC-split it (device+PDS co-sign; standard ECDSA output; heavier crypto dependency).
- **Watch:** #4739 / indigo #1356 progress; any lexicon or DID-doc spec change touching `verificationMethods`; proposals [0013 (consistent DID-service references)](https://github.com/bluesky-social/proposals/tree/main) and [0014 (service-auth audience revised)](https://github.com/bluesky-social/proposals/tree/main), which formalize `did#fragment` key references for non-repo-signing keys.

### 8.2 Sync v1.1 ("inductive firehose") — rolling out; ezpds nearly conformant

- **[Proposal 0006](https://github.com/bluesky-social/proposals/tree/main/0006-sync-iteration)** / [relay updates](https://atproto.com/blog/relay-updates-sync-v1-1): relays are now **non-archival**; each `#commit` is fully validated in sequence via MST inversion. PDS requirements: `prevData` on `#commit`, per-op **`prev`** (previous record CID) on update/delete repoOps, `#sync` events, avoid `tooBig`; `#handle`/`#migration`/`#tombstone` removed in favor of `#identity`/`#account`.
- **ezpds status (verified 2026-07-04):** `prevData` ✅ (nullable tag-42 link, chained across commits), `#sync` ✅ (genesis + head assertions), `#identity` ✅. **Residual:** per-op `prev` is absent from `RepoOp`/`RepoOpWire`/the persisted log — tracked as **MM-234**.

### 8.3 Granular auth scopes (proposal 0011) — shipped upstream; ezpds is behind

- **[Proposal 0011](https://github.com/bluesky-social/proposals/blob/main/0011-auth-scopes/README.md)** defines `repo:`/`rpc:`/`blob:`/`account:`/`identity:` scopes with parameters, plus Lexicon-published **permission sets** (`include:<nsid>`). This is no longer just a proposal: granular scopes + permission sets **went live on bsky.social Aug–Oct 2025** and ship in the reference PDS; `transition:generic`/`transition:email` remain as the legacy bridge. New clients are moving to granular scopes.
- **ezpds status:** advertises only `atproto` + `transition:generic`; app passwords (landed 2026-07-01) use the reference's coarse `com.atproto.appPass[Privileged]` model. Alignment is tracked as **MM-225**, decomposed into **MM-235** (grammar parser + token persistence), **MM-236** (runtime enforcement), **MM-237** (permission sets + consent UI + metadata). The tranquil-style per-scope opt-out consent screen remains the Obsign-differentiator opportunity.
- **Watch:** [permission spec](https://atproto.com/specs/permission) stabilization, [`@atproto/oauth-scopes`](https://www.npmjs.com/package/@atproto/oauth-scopes) as the grammar reference, [discussion #4118](https://github.com/bluesky-social/atproto/discussions/4118).

### 8.4 Other movements worth tracking

- **PLC governance:** Bluesky is moving PLC directory governance to an independent Swiss association — relevant to the wallet's did:plc trust story.
- **Adversarial / self-authorized migration:** the wallet-authorized migration path (wallet holds `rotationKeys[0]` and self-signs the identity leg, no old-PDS cooperation needed — cf. [David Buchanan's adversarial-PDS-migration write-up](https://www.da.vidbuchanan.co.uk/blog/adversarial-pds-migration.html)) is now part of ezpds's own migration design (ADR-0002 / MM-228, MM-229).
- **Proposal 0015 (JSON event stream encoding)** — a possible future alternate firehose encoding; no action needed yet.
- **Re-check cadence:** revisit this section at each phase boundary (or ~quarterly): #4739 status, 0006 rollout deadlines on bsky.network, 0011 spec/`oauth-scopes` drift, PLC governance changes.
