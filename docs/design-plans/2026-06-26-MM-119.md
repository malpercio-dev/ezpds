# MM-119: Iroh Tunnel Integration — Relay Endpoint for Device Connections

**Wave:** 5 (Federation) · **Milestone:** v0.1 — Mobile-Only PDS · **Crate:** `crates/relay/`

## Summary

Stand up a real Iroh QUIC endpoint inside the relay so mobile devices can reach it
through NAT/firewalls by dialing a stable node identity instead of a routable IP. Today
the Iroh surface is a **stub**: `IrohConfig { endpoint: Option<String> }` is a static
config string and `GET /v1/devices/:id/relay` simply echoes it back as `irohEndpoint`.
The relay runs no Iroh endpoint and accepts no Iroh connections.

This change makes it real:

1. Add the `iroh` crate (1.0.x) as a workspace dependency.
2. Boot an Iroh `Endpoint` alongside the axum HTTP server, with a **persistent** node
   identity (secret key stored in the DB, mirroring the OAuth ES256 / JWT key pattern) so
   the published node id is stable across restarts.
3. Run an accept loop on a dedicated ALPN (`ezpds/iroh/0`) that handles a minimal
   bidirectional message protocol (a ping/echo handshake for v0.1 — proves the channel
   end-to-end and gives the device a liveness probe).
4. Publish the **live** node id via `GET /v1/devices/:id/relay` → `irohEndpoint`, sourced
   from the running endpoint rather than a hand-entered config string.
5. Close the endpoint cleanly during graceful shutdown.

The repo-sync / push-notification payloads that will eventually ride this tunnel are **out
of scope** for v0.1 — the AC only requires that the endpoint runs, a device can connect by
node id, and basic message passing works. The echo protocol is the seam those future
protocols slot into.

## Decisions (locked)

1. **Persistent node identity.** Persist the Iroh secret key in SQLite via a
   `load_or_create_iroh_secret_key` helper next to `load_or_create_oauth_signing_key`
   (`auth/signing_key.rs`), wrapped by the existing `signing_key_master_key` when present.
   The published node id is therefore stable across restarts, so devices' cached ids stay
   valid. Costs one migration. (Rejected: ephemeral per-boot key — it would invalidate every
   device's cached id on each restart.)

2. **Publish a bare node id with n0 discovery.** Enable iroh's n0 discovery
   (`.discovery_n0()`) + default relays so a client can resolve addresses from the node id
   alone; publish just the node id string (short, stable). (Rejected for v0.1: `NodeTicket`
   bundling direct addresses — more robust on hostile networks but longer and
   address-dependent; revisit if field holepunching proves unreliable.)

3. **`enabled` flag + existing `endpoint` config string.** Add `IrohConfig.enabled: bool`
   (default `false`) so the endpoint is opt-in for v0.1 and plain `cargo run -p relay` /
   existing tests are unaffected. Keep `IrohConfig.endpoint: Option<String>` (default `None`)
   as an optional manual override — when set, advertise it instead of the live id; otherwise
   advertise the live id. The two existing config-echo tests in `get_device_relay.rs` are
   updated to cover live-id, override, and disabled cases.

## Definition of Done

- Relay boots an Iroh endpoint alongside the HTTP server when `[iroh] enabled = true`; a
  disabled or unconfigured relay behaves exactly as today.
- The endpoint's node identity is stable across restarts.
- `GET /v1/devices/:id/relay` returns the live node id in `irohEndpoint` when Iroh is
  enabled, and omits the field when disabled.
- A client iroh endpoint can connect to the relay by node id over the `ezpds/iroh/0` ALPN
  and complete a ping/echo round-trip (in-process integration test).
- The endpoint is closed on SIGTERM/Ctrl-C before the process exits.
- `just ci-relay` passes (fmt, clippy `-D warnings`, test, audit), including on the Linux
  tangled-spindle CI and the release `Dockerfile` build.
- `bruno/` entry for `get_device_relay` notes the `irohEndpoint` value is now the live id.

## Acceptance Criteria Coverage (from MM-119)

### MM-119.AC1: Relay starts Iroh endpoint alongside HTTP server
- **AC1.1 Success:** With `[iroh] enabled = true`, startup binds the Iroh endpoint and logs
  the node id; the HTTP server still binds and serves.
- **AC1.2 Success:** With Iroh disabled/unset, startup is unchanged (no endpoint, no new
  failure modes).
- **AC1.3 Failure:** If the Iroh endpoint fails to bind while enabled, startup fails loudly
  with context (consistent with the pool/migration error handling in `main.rs`).

### MM-119.AC2: Device can connect via Iroh node id
- **AC2.1 Success:** `GET /v1/devices/:id/relay` returns the running endpoint's node id in
  `irohEndpoint` (authenticated device).
- **AC2.2 Success:** A second iroh endpoint dials that node id and establishes a connection
  on the `ezpds/iroh/0` ALPN.
- **AC2.3 Edge:** The published node id is identical before and after a relay restart
  (identity persistence).

### MM-119.AC3: Basic message passing works
- **AC3.1 Success:** Client opens a bi stream, sends a ping frame, receives the echo/pong.
- **AC3.2 Failure:** A connection on an unknown ALPN is rejected.
- **AC3.3 Edge:** Endpoint close mid-connection terminates the stream without panicking the
  accept loop.

### MM-119.AC4: Works through NAT (local network simulation)
- **AC4.1 Success:** In-process two-endpoint test over loopback completes the ping/echo
  (covers "basic message passing" deterministically in CI).
- **AC4.2 Manual:** Cross-network validation (device on cellular, relay behind NAT) relies on
  iroh's holepunching + n0 relays; documented as a manual smoke test in `docs/`, not a CI
  gate, because real holepunching is non-deterministic in a sandbox.

## Architecture

### New module: `crates/relay/src/iroh.rs` (`// pattern: Imperative Shell`)
Owns all process-level Iroh state and the accept loop. Public surface:
- `IrohState { endpoint: iroh::Endpoint, node_id: String }` — stored in `AppState`.
- `async fn start(secret_key, config) -> anyhow::Result<IrohState>` — builds the endpoint
  (`Endpoint::builder().secret_key(..).alpns(vec![ALPN]).discovery_n0().bind().await`),
  caches the node id string.
- `fn spawn_accept_loop(endpoint) -> JoinHandle<()>` — mirrors `blob_gc::spawn_blob_gc`:
  loops on `endpoint.accept()`, spawns a per-connection task running the echo protocol over
  `connection.accept_bi()`. All errors logged, never propagated (a bad peer never kills the
  loop).
- `const ALPN: &[u8] = b"ezpds/iroh/0";`

The v0.1 protocol is intentionally trivial (read frame → write frame back). It exists to
satisfy AC3 and to be the extension point for the real repo-sync/push protocols later;
those will register additional ALPNs or message types here.

### `AppState` (`app.rs`)
Add `pub iroh: Option<Arc<crate::iroh::IrohState>>` (None when disabled). `get_device_relay`
reads `state.iroh.as_ref().map(|i| i.node_id.clone())` for `irohEndpoint`, falling back to
the optional config override, else omitting the field.

### Secret-key persistence (`auth/signing_key.rs` + migration)
`load_or_create_iroh_secret_key(&pool, master_key)` — load 32-byte Iroh secret from a new
single-row table (e.g. `iroh_identity`), or generate + store on first boot. Follows the
existing load-or-create pattern; encrypt-at-rest with `signing_key_master_key` if present.

### Startup (`main.rs`)
After the keys are loaded and before/after the TCP bind: if `config.iroh.enabled`, load the
secret key, `iroh::start(..).await?`, spawn the accept loop, place `Some(Arc::new(state))`
into `AppState`. Add `endpoint.close().await` to the graceful-shutdown path (the accept-loop
`JoinHandle` is dropped like `_blob_gc`).

### Config (`crates/common/src/config.rs` + `config_loader.rs`)
Extend `IrohConfig`: add `enabled: bool` (default false); keep `endpoint: Option<String>` as
an optional manual override. New env vars: `EZPDS_IROH_ENABLED`; `EZPDS_IROH_ENDPOINT`
already exists. Update `apply_env_overrides` + validation.

## Phases

1. **Config + dependency.** Add `iroh` to workspace `Cargo.toml`; extend `IrohConfig`
   (`enabled`, env wiring); confirm it builds on Linux CI and in the `Dockerfile`. Gate:
   `cargo build -p relay`, config unit tests.
2. **Persistent identity.** Migration + `load_or_create_iroh_secret_key` with master-key

3. **Endpoint + accept loop.** `iroh.rs` (`start`, `spawn_accept_loop`, echo protocol);

4. **Publish + handler.** Source `irohEndpoint` from the live endpoint; update

5. **Integration test + docs.** In-process client↔relay ping/echo test (AC2/AC3/AC4.1);

## Manual validation (NAT smoke test)

CI covers the protocol deterministically over loopback (`iroh_tunnel` tests: echo round-trip
and unknown-ALPN rejection, both offline via the `Minimal` preset). Real NAT traversal —
holepunching through two different networks via n0 relays — is non-deterministic and cannot be
a CI gate, so it is validated manually:

1. Run a relay with the tunnel enabled and a master key set, on a host behind NAT:
   `EZPDS_IROH_ENABLED=true EZPDS_SIGNING_KEY_MASTER_KEY=<64-hex> cargo run -p relay`.
   Note the `Iroh endpoint bound` log line and its `node_id`.
2. From a second machine on a *different* network (e.g. a laptop on cellular/hotspot), run a
   minimal iroh client that dials that `node_id` over the `ezpds/iroh/0` ALPN, opens a bi
   stream, writes a few bytes, and asserts the same bytes come back.
3. Confirm the connection establishes and echoes. Inspect the relay logs for
   `iroh connection accepted`. A direct (holepunched) path is preferred; a relayed path still
   proves reachability through NAT.
4. Restart the relay and re-fetch `GET /v1/devices/:id/relay`; the advertised node id must be
   unchanged (identity persistence).

## Risks / watch-items

- **Dependency weight & CI/Docker build.** `iroh` pulls in `quinn` + a sizable QUIC/crypto
  tree. Must verify the Linux tangled-spindle gate and the release `Dockerfile`
  (`cargo build --release --locked -p relay`) still build and that `cargo audit` stays
  clean. Pin a known-good `iroh` version and run `cargo update -p iroh --precise` if audit
  flags a transitive.
- **API churn.** iroh's API moved across 0.90 → 1.0 (e.g. `NodeId`/`EndpointId` naming,
  `presets`). Pin the exact version and verify method names against that version's docs
  before writing the endpoint code — treat the names in this plan as the 1.0 shape, not
  gospel.
- **`Sync`/`Clone` in `AppState`.** `AppState` is `Clone` and shared across handlers;
  `IrohState` goes behind `Arc`, and `iroh::Endpoint` is already cheaply cloneable/`Send +
  Sync`, so this is fine — but confirm no `!Send` guard is held across an `.await` in the
  accept loop.
- **AAA/quiet-by-default.** Endpoint defaults to disabled so existing deployments and the
  full test suite are untouched until explicitly turned on.
