## End-to-End: Config-to-Endpoint Integration

**Purpose:** Validate that env vars flow correctly into the endpoint.

1. Set environment variables:
   ```
   export EZPDS_ADMIN_TOKEN="manual-test-token"
   export EZPDS_SIGNING_KEY_MASTER_KEY="0102030405060708090a0b0c0d0e0f101112131415161718191a1b1c1d1e1f20"
   ```
2. Start the relay (ensure `data_dir`, `public_url`, and `available_user_domains` are configured).
3. Send a request with no auth:
   ```
   curl -s -o /dev/null -w "%{http_code}" -X POST http://localhost:8080/v1/relay/keys \
     -H "Content-Type: application/json" \
     -d '{"algorithm": "p256"}'
   ```
   Expected: `401`
4. Send a request with the wrong token:
   ```
   curl -s -o /dev/null -w "%{http_code}" -X POST http://localhost:8080/v1/relay/keys \
     -H "Content-Type: application/json" \
     -H "Authorization: Bearer wrong-token" \
     -d '{"algorithm": "p256"}'
   ```
   Expected: `401`
5. Send a valid request:
   ```
   curl -s -X POST http://localhost:8080/v1/relay/keys \
     -H "Content-Type: application/json" \
     -H "Authorization: Bearer manual-test-token" \
     -d '{"algorithm": "p256"}'
   ```
   Expected: HTTP 200 with JSON body containing `keyId` (starts with `did:key:z`), `publicKey` (starts with `z`, no `did:key:` prefix), and `algorithm` equal to `"p256"`. No `privateKey` or `private_key` field.
6. Send a request with unsupported algorithm:
   ```
   curl -s -o /dev/null -w "%{http_code}" -X POST http://localhost:8080/v1/relay/keys \
     -H "Content-Type: application/json" \
     -H "Authorization: Bearer manual-test-token" \
     -d '{"algorithm": "k256"}'
   ```
   Expected: `400`

## End-to-End: Missing Master Key Behavior

**Purpose:** Validate 503 when master key is not configured.

1. Unset the master key: `unset EZPDS_SIGNING_KEY_MASTER_KEY`
2. Keep `EZPDS_ADMIN_TOKEN="manual-test-token"` set.
3. Start (or restart) the relay.
4. Expected: The relay starts successfully (no crash on missing master key).
5. Send a valid auth request:
   ```
   curl -s -o /dev/null -w "%{http_code}" -X POST http://localhost:8080/v1/relay/keys \
     -H "Content-Type: application/json" \
     -H "Authorization: Bearer manual-test-token" \
     -d '{"algorithm": "p256"}'
   ```
   Expected: `503`

## End-to-End: Database Persistence Check

**Purpose:** Confirm the generated key is persisted in SQLite with encrypted private key.

1. After a successful key generation (Config-to-Endpoint step 5), note the `keyId` from the response.
2. Open the relay database: `sqlite3 <data_dir>/relay.db`
3. Run:
   ```sql
   SELECT id, algorithm, public_key, length(private_key_encrypted), created_at
   FROM relay_signing_keys WHERE id = '<keyId>';
   ```
   Expected: One row with `id` matching `keyId`, `algorithm` = `p256`, `public_key` matching response `publicKey`, `length(private_key_encrypted)` = `80`, and `created_at` with an ISO 8601 timestamp.
4. Run:
   ```sql
   SELECT private_key_encrypted FROM relay_signing_keys WHERE id = '<keyId>';
   ```
   Expected: An 80-character base64 string (letters, digits, `+`, `/`, possibly `=` padding). Must not look like raw hex or plaintext.
