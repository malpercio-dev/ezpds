# User & Operator Documentation Sites Design

Tracking issue: [MM-357](https://linear.app/malpercio/issue/MM-357)

Status: **landed** — the Astro Starlight docs sites + independent Railway deploy shipped in #276 (MM-359), and the generated reference pages in #280 (MM-360).

## Summary

The repo already ships a static marketing site (`sites/marketing/`) split into two
audience-scoped surfaces — Obsign (user) and Custos (operator) — each in its own
design register. This design extends that split into two **documentation** surfaces
that explain every feature, screen, route, and operator config, and — crucially —
stay current without relying on anyone remembering to update them.

The anti-drift strategy reuses patterns the repo already trusts. The Bruno route
parity gate (`just bruno-check`) and the harness's command-coverage rule
(browser-harness.AC1.3) both prove the same idea works here: **derive documentation
from the source of truth wherever a machine can, and gate the rest in CI so drift
cannot merge.** Reference pages (HTTP/XRPC API, operator config/env, IPC command
surface) are generated from the same registries those existing checks read; a new
`just docs-check` parity gate joins the `just ci` aggregate and fails the build when a
route, config field, or command exists with no documentation entry.

Screenshots are generated by driving the **browser test harness**
([2026-07-12-browser-harness.md](2026-07-12-browser-harness.md), MM-324): because that
harness boots both mobile app frontends in a plain (Linux-runnable) browser with
deterministic, named scenario presets — including rare error states — a Playwright
script can capture every UI state reproducibly, so doc/marketing imagery cannot go
stale silently.

The release process gains two documentation steps: a per-PR **changelog fragment**
habit that `set-version` rolls into a dated `CHANGELOG.md`, and a release-time
**docs/marketing review pass** (automatable as a Claude Code Routine that diffs the
release range, regenerates derived docs + screenshots, and opens a PR that rides the
normal parity gates).

Documentation tooling is **Astro Starlight**: it reuses the existing node 22 / pnpm /
vite toolchain, provides search and per-version docs out of the box, and can consume the
forked OKLCH token layer the marketing site already established.

## Definition of Done

1. Two documentation surfaces exist and deploy independently — a **user** surface
   (Obsign register) and an **operator** surface (Custos register) — served as their
   own Railway static service, scoped to `sites/docs/` exactly as `sites/marketing/`
   is scoped, with no dependency on the repo-root PDS build.
2. The high-drift reference surfaces are **generated from the source of truth**, not
   hand-transcribed: the HTTP/XRPC API reference, the operator config/env reference, and
   each app's IPC command list.
3. A `just docs-check` parity gate is part of `just ci`/`ci-pds` and fails when a
   registered route, a config field, or an IPC command has no documentation entry —
   drift is a red check, not a hope.
4. App screenshots in the docs (and available to the marketing site) are generated by
   Playwright-driving the browser harness across its named scenarios, regenerated on
   every docs build so they track the real UI.
5. `CHANGELOG.md` exists (Keep a Changelog format), fed by per-PR fragments in
   `changelog.d/`; `just set-version` rolls fragments into a dated release section, and a
   CI gate requires a fragment on PRs that change shipped surfaces.
6. The release runbook documents the documentation steps, and a Claude Code Routine can
   perform the release-time docs/marketing review pass (diff the range → regenerate
   derived docs + screenshots → draft prose → open a PR).

Out of scope (deliberately): migrating the internal `docs/` tree (specs, ADRs, design
plans) into the published sites — those remain raw material, not the published product;
localization/translation of the docs; and iOS-simulator/device screenshots (the harness
is explicitly a browser surface — device imagery stays a separate effort).

## Acceptance Criteria

### docs.AC1: Two documentation surfaces exist and deploy independently
- **docs.AC1.1 Success:** `sites/docs/` builds a user surface (Obsign register) and an
  operator surface (Custos register) with working nav + search.
- **docs.AC1.2 Success:** The docs site deploys as its own Railway service scoped to
  `sites/docs/` (own build config), independent of the repo-root `railway.toml`, mirroring
  the marketing-site service in `docs/deploy.md`.
- **docs.AC1.3 Success:** The two registers do not cross-apply: the user surface uses the
  Obsign token register and the operator surface uses the Brass Console register, forked
  from the live app token layers (same discipline as `sites/marketing/README.md`).
- **docs.AC1.4 Success:** Held invariants match the marketing site — WCAG 2.2 AAA targets,
  self-hosted fonts (no runtime CDN), status never by color alone.

### docs.AC2: Reference pages are generated, not transcribed
- **docs.AC2.1 Success:** The HTTP/XRPC API reference is generated from the same route
  registry `scripts/bruno-parity.sh` reads; regenerating after a route change updates the
  page with no manual edit.
- **docs.AC2.2 Success:** The operator config/env reference is generated from the PDS
  configuration type(s) (serde/doc-comment reflection), so a new config field appears in the
  docs from its Rust definition.
- **docs.AC2.3 Success:** Each app's IPC command surface is listed, derived from the app's
  `ipc.ts` command registry (the same surface the harness registry enumerates).

### docs.AC3: Drift is gated in CI
- **docs.AC3.1 Success:** `just docs-check` fails when a registered route has no API-reference
  entry (analogue of `just bruno-check`).
- **docs.AC3.2 Success:** `just docs-check` fails when a config field has no config-reference
  entry.
- **docs.AC3.3 Success:** `just docs-check` is wired into `just ci` and `just ci-pds`.
- **docs.AC3.4 Success:** The version/build stamp shown in the docs is derived from
  `Cargo.toml` `[workspace.package].version` (the same source `_health/describeServer` reports),
  so docs version and server version cannot diverge.

### docs.AC4: Screenshots are generated from the harness
- **docs.AC4.1 Success:** `just docs-screenshots` Playwright-drives the browser harness in
  fake mode, iterating its named scenarios and capturing each screen into `sites/docs/`.
- **docs.AC4.2 Success:** Error-state and rare-state screenshots (e.g. the harness's
  `migration-in-flight`, `degraded-health`, `failNext`-injected error states) are captured, not
  just happy paths.
- **docs.AC4.3 Success:** Regenerating after a UI change produces changed images; the recipe is
  deterministic (no reliance on `Date.now()`/network).
- **docs.AC4.4 Edge:** The recipe runs on a Linux CI runner (no macOS/Tauri needed), consistent
  with the harness running the app frontend without the native shell.

### docs.AC5: Changelog is a per-PR habit
- **docs.AC5.1 Success:** `CHANGELOG.md` exists in Keep a Changelog format with a released
  history and no permanently-churning `Unreleased` block.
- **docs.AC5.2 Success:** A PR touching shipped surfaces without a `changelog.d/` fragment fails
  a CI gate (analogue of the Bruno rule).
- **docs.AC5.3 Success:** `just set-version X.Y.Z` rolls `changelog.d/` fragments into a dated
  `## [X.Y.Z]` section and clears the fragment directory.

### docs.AC6: Release-time docs pass
- **docs.AC6.1 Success:** The release runbook (`docs/deploy.md`) documents the documentation
  steps in order: roll changelog → regenerate derived docs + screenshots (gates green) →
  docs/marketing review pass.
- **docs.AC6.2 Success:** A Claude Code Routine exists that, given a release range, regenerates
  derived docs + screenshots, drafts changelog/doc/marketing prose, and opens a PR that rides the
  normal parity gates — a human reviews the PR rather than authoring from scratch.

## Glossary

- **Documentation surface**: One audience-scoped body of docs. There are two — user
  (Obsign) and operator (Custos) — mirroring the marketing site's existing split.
- **Design register**: A scoped visual system. Obsign ("Sealed Credential") vs the admin
  companion's "Brass Console"; the repo forbids cross-applying them (root `AGENTS.md` → Design
  Context).
- **Astro Starlight**: A documentation static-site framework on the Astro build tool. Chosen
  because it reuses the existing node/pnpm/vite toolchain and provides search + versioned docs.
- **Parity gate**: A CI script that fails when two surfaces that must agree have drifted. The
  repo already ships several (`bruno-check`, `font-check`, `cap-check`, `ios-paths-check`); this
  design adds `docs-check`.
- **Source of truth (for a reference page)**: The single in-repo definition a doc page is
  generated from — the route registry for the API reference, the config type for the config
  reference, `ipc.ts` for the command surface, `Cargo.toml` for the version.
- **Changelog fragment**: A small per-PR file in `changelog.d/` describing one user-visible
  change (towncrier-style), rolled into `CHANGELOG.md` at release. Avoids merge-conflict churn on
  a shared `Unreleased` block.
- **Browser (test) harness**: The MM-324 harness that boots each mobile app frontend in a plain
  browser with deterministic, scriptable scenarios; the screenshot engine for this design.
- **Scenario / `window.__harness`**: The harness's named state presets and runtime control API;
  the screenshot recipe drives them to reach each documented UI state.
- **Derived docs**: Doc pages produced by a generator from a source of truth, as opposed to
  hand-authored prose. Only derived docs are covered by `docs-check`.
- **Docs/marketing review pass**: The release-time step that decides which hand-authored docs and
  marketing pages need edits for the release, and drafts them.
- **Claude Code Routine**: A scheduled/triggered Claude Code (web) session; the vehicle for
  automating the review pass.

## Architecture

### Site layout

`sites/docs/` is a new Astro Starlight project, deployed as a second static Railway service
scoped by `Root Directory = sites/docs/` (own `Dockerfile`/`Caddyfile`/`railway.toml`), exactly
as `sites/marketing/` is a service independent of the repo-root PDS build. Two top-level
sections — **User** (Obsign register) and **Operator** (Custos register) — each import the
forked token layer for their register. Held invariants (WCAG 2.2 AAA, self-hosted fonts, no CDN,
status never by color alone) carry over from the marketing site.

The internal `docs/` tree stays where it is and is **not** published; it is the raw material
(specs, ADRs, design plans) that curated user/operator pages are written from.

### Content classes

Three classes of page, handled differently:

1. **Hand-authored guides** — onboarding, recovery, Shamir backup, migration, running a relay,
   moderation. Human prose; covered by the release-time review pass, not by `docs-check`.
2. **Generated reference** — API reference, operator config/env reference, IPC command surface.
   Produced by generators from a source of truth; covered by `docs-check` parity gates.
3. **Generated media** — app screenshots per harness scenario. Produced by `just docs-screenshots`.

### Generators and the parity gate

Following the `bruno-parity.sh` model, each generated-reference page has (a) a generator that
emits the page from the source of truth and (b) a check that the emitted page and the source
agree:

- **API reference** ← the route registry that `scripts/bruno-parity.sh` already enumerates.
  `docs-check` reuses that enumeration: every registered route must appear in the reference.
- **Config/env reference** ← the PDS config type(s). A generator reflects serde field names +
  doc-comments into markdown; `docs-check` fails on an undocumented field.
- **IPC command surface** ← each app's `ipc.ts` registry (the same list the harness registry is
  typed against). A command with no doc entry fails the check.
- **Version stamp** ← `Cargo.toml` `[workspace.package].version`, injected at build.

`just docs-check` runs these checks and joins the `just ci` / `just ci-pds` aggregate next to
`bruno-check`, `font-check`, `cap-check`, `ios-paths-check`.

### Screenshot pipeline

`just docs-screenshots` starts the harness dev server (`VITE_HARNESS=fake`) for each app, then a
Playwright script iterates the scenario list (`window.__harness.scenario(...)`, plus
`failNext(...)` for error states), navigates each screen, and writes deterministic PNGs into
`sites/docs/`. Because the harness runs the frontend without the native shell, this runs on a
Linux runner. Optionally, a visual-diff step flags unexpected image changes on PRs.

### Release integration

- **Per PR:** author drops a `changelog.d/<id>.<type>.md` fragment; a CI gate requires one on
  PRs that change shipped surfaces.
- **At `just set-version`:** roll fragments into a dated `## [X.Y.Z]` section of `CHANGELOG.md`,
  clear `changelog.d/`.
- **At release:** regenerate derived docs + screenshots (gates green), then run the docs/marketing
  review pass. Docs deploy like marketing — `main` → staging docs; a `vX.Y.Z` snapshot pins
  versioned docs.

### Automation ladder

- **Level 0 — discipline:** `docs-check` parity gates + the per-PR changelog fragment. Nothing
  is generated, but nothing rots silently. Highest value per unit effort; land first.
- **Level 1 — generation:** derived reference pages + harness screenshots rebuilt every docs
  build. Removes the bulk of manual upkeep.
- **Level 2 — agentic:** a release-time Claude Code Routine drives Level 1, drafts prose, and
  opens a PR reviewed by a human. The parity gates are the safety net that keeps the agent honest.

## Existing Patterns

- **Audience-scoped surfaces** — `sites/marketing/{index.html, custos/index.html}` already split
  user vs operator by design register; the docs sites mirror this rather than inventing a taxonomy.
- **Independent static Railway service** — the marketing site is already a second Railway service
  scoped by Root Directory with its own Caddy/Dockerfile; the docs site copies that scaffold.
- **Forked token layer** — `sites/marketing/assets/css/site.css` is forked from the live app token
  layers with a "must not drift, re-fork on brief change" rule; the docs site reuses that approach.
- **Parity gate as anti-drift** — `just bruno-check` (route ⇄ Bruno collection) and the harness's
  command-coverage rule (browser-harness.AC1.3) are the direct precedent for `docs-check`.
- **Version as single source of truth** — `Cargo.toml` `[workspace.package].version` already flows
  to `_health/describeServer`; the docs version stamp reads the same field so it cannot diverge.
- **Harness scenarios** — the browser harness's deterministic named presets and `window.__harness`
  control API are the screenshot engine; this design consumes them rather than adding a new one.
- **`just ci` aggregate** — the docs gates slot into the same local/CI gate that already runs the
  other parity checks.

## Additional Considerations

**Don't publish the internal `docs/` tree.** Specs, ADRs, and design plans are the source material
for curated pages, not the published product. Conflating them would leak in-flight/internal detail
and couple doc structure to internal doc structure.

**What the gates can and cannot catch.** `docs-check` enforces *coverage* (every route/config
field/command has an entry) — like `bruno-check`, it does not verify the *prose* is correct. Prose
accuracy stays a review-pass concern; the generators shrink the hand-authored surface so review has
less to get wrong.

**Screenshot fidelity boundary.** Harness screenshots inherit the harness's own caveat: they are
browser renders, not device renders (no real keychain/Secure-Enclave, biometric prompt,
WKWebView-specific rendering, safe-area insets). Good enough for feature documentation; not a
substitute for device/simulator imagery, and the docs should say so where it matters.

**Fragment gate scoping.** The changelog-fragment requirement must scope to shipped surfaces (like
the Bruno rule targets registered routes), so docs-only or internal-refactor PRs are not forced to
invent a user-facing note.

**Sequencing.** Phase 1 (changelog) and the Level-0 parity gates deliver the most durability per
unit effort and have no dependency on the harness; do them first. The agentic Phase 5 lands last,
once the deterministic generators are trustworthy enough to let an agent lean on them.
