# MM-143: Tauri Mobile Project Scaffolding Design

## Summary

This document specifies the scaffolding for `apps/identity-wallet/`, the first mobile application in this repository. The app is built with Tauri v2 — a framework that packages a Rust binary and a web frontend together into a native mobile app — using SvelteKit 2 with Svelte 5 as the frontend and a Rust crate as the backend. On iOS, the frontend runs inside a native WKWebView and communicates with the Rust backend exclusively through Tauri's IPC bridge, which lets TypeScript code call Rust functions by name via `invoke()`. The frontend is built as a fully static bundle (no server-side rendering) so it can be embedded in the native app without a web server.

The work is structured in three phases: first, establishing the project directory structure and integrating the new `src-tauri` crate into the existing Cargo workspace so the entire repo continues to build cleanly; second, wiring up Tauri's configuration, the `greet` IPC command, and the SvelteKit page that exercises it end-to-end in an iOS simulator; and third, adding the new mobile tooling (`cargo-tauri`, Node.js, pnpm) to the Nix dev shell and documenting the iOS workstation setup for developers. Automated CI covers Rust and frontend checks but excludes simulator testing, which requires a macOS machine with Xcode and is run manually.

## Definition of Done

1. `apps/identity-wallet/` exists with a SvelteKit 2 + Svelte 5 frontend (Vite, `@sveltejs/adapter-static`) and a Tauri v2 Rust backend (`src-tauri/`).
2. `apps/identity-wallet/src-tauri` is a member of the root Cargo workspace; `cargo build` at workspace root succeeds cleanly.
3. `cargo tauri ios dev` launches the app in an iOS simulator displaying a placeholder screen.
4. IPC bridge works: the SvelteKit frontend calls a `greet(name)` Rust command and displays the response.
5. `devenv.nix` includes `cargo-tauri`, Node.js, and pnpm; developer setup for Xcode, iOS simulators, and Cocoapods is documented.
6. A suggested CI pipeline spec is documented (two parallel jobs: `rust-check` and `frontend-check`); implementation in tangled.org CI is left to the developer.

## Glossary

- **Tauri v2**: A framework for building desktop and mobile applications with a Rust backend and a web frontend. The Rust side exposes commands; the frontend runs in a native WebView and calls those commands over an IPC bridge.
- **IPC bridge**: Inter-process communication channel provided by Tauri. The frontend calls `invoke('command_name', args)` in TypeScript; Tauri routes the call to the matching `#[tauri::command]` Rust function and returns the result.
- **`invoke()`**: The TypeScript function from `@tauri-apps/api/core` used to call Rust commands across the IPC bridge.
- **`#[tauri::command]`**: A Rust attribute macro that marks a function as callable from the frontend via `invoke()`. The function must be registered with `generate_handler!` in the Tauri builder.
- **`generate_handler!`**: A Tauri macro that produces the dispatch table mapping command name strings (received over IPC) to their Rust handler functions.
- **SvelteKit 2**: A full-stack web framework built on Svelte. Used here only as a frontend build system; SSR features are disabled.
- **Svelte 5**: The Svelte component compiler version. Introduces "runes" — a reactivity primitive syntax (`$state`, `$derived`) that replaces the implicit reactive declarations of Svelte 4.
- **Runes**: Svelte 5's explicit reactivity model. `$state()` declares reactive variables; `$derived()` declares computed values. Used in `+page.svelte` for the Greet button demo.
- **adapter-static**: A SvelteKit build adapter that outputs a fully static HTML/JS/CSS bundle with no server component. Required for Tauri because the WebView loads files from disk, not from a running server.
- **SSR (Server-Side Rendering)**: Generating HTML on a server before sending it to the client. Disabled here (`ssr = false`) because Tauri apps have no web server — the frontend runs entirely in the native WebView.
- **WKWebView**: Apple's iOS/macOS WebView component. Tauri uses it to render the frontend HTML/JS inside the native app shell.
- **HMR (Hot Module Replacement)**: Vite's development feature that pushes code changes to the browser without a full page reload. In Tauri iOS dev, the simulator connects to the Vite dev server over LAN, so `hmr.host` must be set to the machine's LAN IP (not `localhost`).
- **`vite.config.ts`**: Configuration file for Vite, the build tool and dev server used by SvelteKit. The Tauri-specific settings here (host, port, HMR config) are required for the iOS simulator to connect to the dev server.
- **`tauri.conf.json`**: The primary Tauri configuration file. Specifies the app bundle ID, window settings, the URL Tauri loads during development (`devUrl`), and the directory of the production static build (`frontendDist`).
- **`build.rs`**: A Cargo build script. Tauri requires one that calls `tauri_build::build()` to perform code generation steps before the main crate compiles.
- **`crate-type = ["staticlib", "cdylib", "rlib"]`**: A Cargo setting that compiles the crate in multiple link formats. `staticlib` is required for iOS, `cdylib` for Android, and `rlib` enables normal `cargo build` and tests without a mobile target.
- **Cargo workspace**: Cargo's mechanism for managing multiple Rust crates in one repository with a shared `Cargo.lock` and shared dependency versions. The new `src-tauri` crate is added as a workspace member so it builds alongside the existing relay, repo-engine, crypto, and common crates.
- **`devenv.nix`**: The Nix devenv configuration file for this repository. Adding packages here makes them available in the developer shell after running `nix develop`.
- **Cocoapods**: A dependency manager for Apple platform libraries. Tauri's iOS build process uses it to link native Apple frameworks; developers must install it separately from Xcode.
- **`cargo tauri ios init`**: A one-time CLI command that generates the Xcode project in `src-tauri/gen/apple/`. Must be re-run per developer because the output is machine-specific and is gitignored.
- **`cargo tauri ios dev`**: The CLI command that starts the Vite dev server, compiles the Rust backend, and launches the app in the iOS simulator with live reload.
- **`aarch64-apple-ios` / `aarch64-apple-ios-sim`**: Rust compilation targets for physical iOS devices and Apple Silicon iOS simulators, respectively. Added to `rust-toolchain.toml` so Cargo can cross-compile for iOS.
- **tangled.org CI**: The CI system used by this project. Wiring up the documented pipeline jobs is left to the developer and is outside the scope of this ticket.
- **pnpm**: A Node.js package manager used for the frontend. Chosen here as the package manager for the `apps/identity-wallet/` project.

## Architecture

Tauri v2 mobile app at `apps/identity-wallet/` with a SvelteKit 2 + Svelte 5 frontend and a Rust backend crate. The frontend runs inside a native WebView (WKWebView on iOS); it communicates with the Rust backend exclusively through Tauri's IPC bridge via `invoke()`. No SSR — adapter-static outputs a static bundle the WebView loads.

### Project Structure

```
apps/
  identity-wallet/
    package.json              # pnpm project root
    pnpm-lock.yaml
    svelte.config.js          # adapter-static (SPA mode, fallback: index.html)
    vite.config.ts            # clearScreen, host, port, hmr, envPrefix
    tsconfig.json
    src/
      routes/
        +layout.ts            # ssr = false; prerender = false
        +layout.svelte        # app shell
        +page.svelte          # splash/placeholder + IPC demo
      lib/
        ipc.ts                # typed invoke() wrappers
    src-tauri/
      Cargo.toml              # workspace member; Tauri deps declared locally
      tauri.conf.json         # bundle id, window config, devUrl, frontendDist
      build.rs
      src/
        lib.rs                # #[tauri::command] greet + pub fn run()
        main.rs               # entry point
      gen/                    # gitignored — Tauri-generated Xcode project
        apple/
```

### SvelteKit + Tauri Configuration

**`svelte.config.js`** — SPA mode via adapter-static:

```js
// fallback: 'index.html' routes 404s to index for client-side navigation
adapter: adapter({ fallback: 'index.html' })
```

**`src/routes/+layout.ts`** — disable SSR globally (WebView has no server):

```ts
export const ssr = false;
export const prerender = false;
```

**`vite.config.ts`** — Tauri-required server settings:

- `clearScreen: false` — surfaces Rust compiler errors
- `server.host: '0.0.0.0'`, `port: 5173`, `strictPort: true`
- `hmr.host: await internalIpV4()` — iOS simulator connects via LAN IP, not localhost
- `envPrefix: ['VITE_', 'TAURI_']`

### IPC Bridge

**Rust side (`src-tauri/src/lib.rs`):**

```rust
// Contract only — no function body in design plans
#[tauri::command]
fn greet(name: String) -> String

pub fn run()  // configures builder, registers greet via generate_handler!
```

**TypeScript side (`src/lib/ipc.ts`):**

```ts
import { invoke } from '@tauri-apps/api/core';

export const greet = (name: string): Promise<string> =>
  invoke('greet', { name });
```

### Cargo Workspace Integration

Root `Cargo.toml` gains one `members` entry:

```toml
"apps/identity-wallet/src-tauri"
```

`src-tauri/Cargo.toml` inherits `version`, `edition`, `publish` from the workspace root. All Tauri-specific dependencies (`tauri = "2"`, `tauri-build = "2"`) are declared locally — no other workspace crate needs them, so they stay out of `[workspace.dependencies]`.

```toml
[lib]
crate-type = ["staticlib", "cdylib", "rlib"]
# staticlib → iOS static binary
# cdylib    → Android shared library
# rlib      → normal cargo build / integration tests
```

`rust-toolchain.toml` gains two iOS targets:
- `aarch64-apple-ios` — physical iOS device
- `aarch64-apple-ios-sim` — Apple Silicon simulator

### Nix Dev Environment

`devenv.nix` gains three packages:

```nix
pkgs.cargo-tauri       # tauri-cli (available in nixpkgs)
pkgs.nodejs_22         # Node.js runtime for frontend tooling
pkgs.nodePackages.pnpm # package manager
```

No new devenv processes or environment variables. `cargo tauri ios dev` is run manually by the developer (requires Xcode, cannot be a background process).

`src-tauri/gen/` is added to `.gitignore` — Tauri regenerates this directory on `cargo tauri ios init` and it is machine-specific.

### Developer Setup Documentation

**`apps/identity-wallet/CLAUDE.md`** (new) — comprehensive iOS developer workstation guide:

- Prerequisites: macOS Ventura+, Xcode (latest, from App Store), iOS Simulator platform (Xcode → Settings → Platforms), Cocoapods (`sudo gem install cocoapods`), Apple Developer account (optional for simulator, required for device)
- Dev shell: `nix develop --impure --accept-flake-config`
- First-time frontend setup: `pnpm install` in `apps/identity-wallet/`
- First-time iOS setup: `cargo tauri ios init` (run once; generates `src-tauri/gen/apple/`)
- Development: launch Xcode first (license acceptance), then `cargo tauri ios dev`
- Workspace build without Xcode: `cargo build` from repo root works on any machine in the dev shell

Root `CLAUDE.md` gains a pointer to `apps/identity-wallet/CLAUDE.md` under a "Mobile" section.

### Suggested CI Pipeline

Not implemented — developer will wire up in tangled.org CI. Two parallel jobs:

**`rust-check`:**
1. Install Rust stable with `rustfmt` + `clippy` + iOS targets (`aarch64-apple-ios`, `aarch64-apple-ios-sim`)
2. Cache Cargo registry + `target/` (key: `Cargo.lock`)
3. `cargo fmt --all --check`

4. `pnpm build` (SvelteKit static build via adapter-static)

Mobile simulator testing (`cargo tauri ios dev`) is excluded from automated CI — it requires a macOS runner with Xcode and is run manually.

## Existing Patterns

**Cargo workspace:** Follows the existing workspace resolver v2 pattern. `src-tauri` inherits `version`, `edition`, and `publish` from `[workspace.package]`, matching how all four existing crates are configured. Tauri-specific deps stay local — consistent with how relay-specific deps (axum, clap) are not in `[workspace.dependencies]`.

**devenv.nix:** Follows the existing pattern of adding Nix-provided tools to `packages`. No new patterns introduced — `cargo-tauri`, `nodejs_22`, and `pnpm` are added the same way `just` and `cargo-audit` are.

**CLAUDE.md hierarchy:** Follows the existing `nix/CLAUDE.md` pattern of domain-level CLAUDE.md files for subsystem-specific developer guidance. `apps/identity-wallet/CLAUDE.md` is the mobile equivalent.

**rust-toolchain.toml:** Extends the existing file (no new file) by appending iOS targets alongside the existing macOS and Linux targets.

No existing mobile or frontend patterns to follow — this is the first frontend code in the repository.

## Additional Considerations

**Cargo.lock placement:** Tauri tooling expects `Cargo.lock` at the workspace root (where `Cargo.toml` is). This is already the case in this repo. Do not create a second `Cargo.lock` in `src-tauri/`.

**`src-tauri/gen/` is gitignored:** `cargo tauri ios init` generates a full Xcode project in `src-tauri/gen/apple/`. This is machine-specific and must be regenerated per developer. Committing it causes merge conflicts and bloats the repo.

**Android target:** Excluded from this scaffolding. When Android support is added, `aarch64-linux-android` and `armv7-linux-androideabi` targets plus Android SDK configuration will be needed.

**`[profile.release]` for mobile:** Add `strip = true`, `lto = true`, `opt-level = "z"` to `src-tauri/Cargo.toml` — Tauri's recommendation for reducing iOS binary size. Has no effect on `cargo build --workspace` debug builds.
