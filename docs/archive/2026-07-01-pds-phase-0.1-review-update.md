## 1. Goal — re-confirmed, unchanged

Per [`unified-milestone-map.md`](../unified-milestone-map.md) §2.1: **v0.1 = Mobile-Only PDS. The PDS is a full PDS — hosts the repo, serves XRPC, signs commits (per-account P-256 keys, server-side), emits the firehose.** The user creates an ATProto identity from the iPhone (wallet holds the `did:plc` rotation key; 2-of-3 Shamir recovery split) and logs into Bluesky. Signing on the user's own hardware (the Mac app) is explicitly **v2.0+**.

## 4. Competitive analysis — refresh (what changed since 6/30)

Full detail and per-implementation profiles in the 6/30 doc §4; this section records what's new or corrected.

### cocoon (Go, haileyok) — now the pace-setter on protocol currency

v0.10 (2026-06-15), active monthly cadence. **Shipped Sync v1.1 (`prevData` + `#sync`) in June 2026**; v0.10 added granular OAuth scope parsing + PAR scope validation, repo-write scope enforcement, RFC 7009 revocation. Ran a `recommit-repos` repair migration for previously-invalid revs — evidence that relay-side validation pressure is real. Still explicitly "not production ready," but the author's main account has run on it for months. Known interop issues: strict-integer lexicon validation 500s on third-party records with floats (#62), OAuth PAR 502 with the Heron client (#119), large-repo `importRepo` timeouts (#57).

### tranquil (Rust, tranquil.farm) — the closest architectural cousin, now packaged in nixpkgs

0.6.5, very active (≥5 authors), **upstream NixOS module `services.tranquil-pds` in nixpkgs**; AGPL. Confirmed feature superset: passkeys/WebAuthn + TOTP + backup codes, SSO, did:web accounts, granular scopes with per-scope consent opt-out, scoped app passwords, account delegation with audit log, multi-channel notifications (email/Discord/Telegram/Signal — dedicated crate), built-in web UI + web migrator, experimental **embedded storage engine ("tranquil-store": Bitcask-style blockstore + LSM metastore + segmented firehose log)** motivated by "reference PDS hits structural limits ~300k accounts," and nascent QUIC gossip clustering ("ripple"). Deliberately spec-strict: no default AppView routing (clients must send `atproto-proxy`), no mixing transitional+granular scopes — it breaks clients the reference tolerates. One open OAuth interop bug (stream.place iOS in-app browser). Distributes via ATCR (atproto-native container registry) + GHCR.

### Landscape shifts (new since 6/30 research)

- **Hubble** (June 2026, microcosm + Bluesky): real-time authenticated public mirror of every repo in the atmosphere; users can download their archive even if their PDS dies. Positioned as the safety net for the alt-PDS ecosystem — changes the backup story for every small host (https://hubble.microcosm.blue/).
- **Migration-as-product is a proven lane:** Airport (atpairport.com wizard) and PDS MOOver (migration UI + scheduled S3 backups + missing-blob recovery) both target non-experts; reference PDS still only offers the raw API dance via goat. Since Sept 2025, **bsky.social accepts incoming migrations back**, making bidirectional migration table stakes.
- **Serverless single-user PDS is the fastest-growing lane:** cirrus (Cloudflare Workers/DO/R2, 413 stars in ~6 months, passkeys, migration-tested) and alteran (Workers/D1/R2, per-account blob quotas). Signal: demand for near-zero-cost sovereign hosting.
- **rsky-pds** (Blacksky) remains the only alt PDS with a production multi-tenant user base (Postgres+S3+Mailgun); NLnet-funded.
- millipds's issue #12 divergence list remains the best public checklist of reference-PDS behavioral quirks (Range on `getBlob`, `swapRecord` in `applyWrites`, base64 DAG-CBOR bodies, UNIX-socket serving — all cheap wins).
- Corrections/notes: "bigsky" is the relay (not a PDS); "hepa" is the automod daemon; "parakeet" is an AppView; atompds (C#) archived May 2026 — the C# lane moved to threddyrex's dnproto/rustproto.

### Protocol changes a 2026 PDS must track (from the reference research)

1. **Sync v1.1** (May 2025 rollout; bsky.network cutover 2026-01-27; strict MST validation coming, new hosts held immediately) → MM-223.
2. **OAuth as primary auth**: PRM endpoint, PAR required, DPoP, granular scopes/permission sets (Aug–Oct 2025), `transition:email` → MM-224/225. App passwords deprecated-track, **no announced shutdown date** — keep them (we do).
3. **Lexicon resolution via DNS + repos** (`_lexicon.` TXT → `com.atproto.lexicon.schema` records) — no PDS server changes needed.
4. **PLC directory → independent Swiss association** (Sept 2025); same API; mirror service on roadmap.
5. **Relays are non-archival** — expect direct backfill traffic on `sync.*`; `getRepo`-as-redirect proposals.
6. **IETF standardization** in progress (drafts Sept 2025) — expect normative tightening.

## 5. Net-new feature candidates (recorded, not filed)

Updated candidate list beyond reference parity, ranked by fit with the Obsign thesis:

1. **Per-scope consent UI + scoped app passwords** (tranquil; now partially spec'd upstream as permission sets) — lands naturally while doing MM-225; flagship "see and control exactly what touches your identity" feature.
2. **Passkey/WebAuthn session auth** (tranquil, cirrus) — ezpds is already device-key-centric in provisioning; promote the Secure-Enclave key to a first-class `createSession`/OAuth factor.
3. **In-wallet guided migration wizard** (Airport/MOOver/tranquil web migrator prove demand) — once MM-207 lands, wrap it in Obsign onboarding with honest custody messaging.
4. **Hubble-aware backup/restore story** — advertise "even if this PDS dies, your repo is mirrored and importable"; pairs with existing Litestream + Shamir custody. Possibly ship a one-click "verify my Hubble mirror" check.
5. **Account delegation / guardian model with audit log** (tranquil) — synergistic with Shamir recovery custody.
6. **S3/CDN blob offload with `getBlob` 302 redirect + in-binary scheduled backups** (cocoon, rsky) — keeps the Railway box off the media path.
7. **millipds #12 niceties**: HTTP Range + Content-Disposition on `getBlob`, `swapRecord` in `applyWrites` (verify ours), UNIX-socket serving.
8. **Prometheus/OTel federation-health metrics** (cocoon; we already have OTel tracing — add relay-connectivity/event-throughput metrics).
9. **Multi-channel security alerts** (tranquil: Discord/Telegram/Signal) for "PLC op requested," "new device paired."
10. **did:web option** (tranquil, threddyrex) with honest key-loss caveats (cirrus's framing).
11. **Per-account storage quotas** — ✅ already implemented (ezpds is ahead of the reference here; alteran is the only other one).

## 6. Top 3 recommendations (updated)

### #1 — Sync v1.1 compliance now (MM-223), before relays ratchet validation
This replaces the 6/30 #1 (handle-change loop — since shipped). The relay side of the network changed underneath phase 0.1: `#sync` + `prevData` + inductive validation are the new bar, Bluesky has said enforcement will tighten, and **new hosts get no grace**. ezpds's firehose architecture (durable sequencer, staged transactional emit) makes this a well-scoped addition, and MM-220 (importRepo) can't be finished correctly without it. Highest blocking-risk-per-effort in the backlog.

### #2 — Finish the migration epic (MM-207: MM-219 → 218 → 220 → 222), with MM-211 email as an explicit dependency
Unchanged in substance from 6/30, sharpened in sequencing: `reserveSigningKey` is already in progress; `createAccount`-with-existing-DID and `importRepo`/`listMissingBlobs` are the data legs; the PLC-signing surface is the identity leg and **requires working outbound email** (MM-211) for `requestPlcOperationSignature`. Design it around ezpds's differentiator — the wallet already holds the rotation key client-side, so ezpds can offer migration where *the user*, not the PDS, authorizes the PLC op. bsky.social accepting inbound migrations makes "leave for ezpds, return anytime" a credible, testable loop.

### #3 — OAuth discovery + granular scopes as one arc (MM-224 → MM-225), turning parity into product
MM-224 is a half-day fix that unblocks every spec-strict OAuth client. MM-225 is bigger but is where parity becomes product: granular scopes with a human-readable, per-scope consent UI (tranquil's best idea, now spec-backed as permission sets) doubles as the Obsign flagship control surface, and scoped credentials retroactively upgrade the just-shipped app passwords. Do MM-224 immediately; schedule MM-225 behind the migration epic.
