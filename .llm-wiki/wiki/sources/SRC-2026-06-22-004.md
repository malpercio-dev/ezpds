# Crypto Crate Architecture (CLAUDE.md)

## Summary

API documentation for the [[entities/crypto|Crypto Crate]] — a pure Functional Core providing cryptographic primitives for the ezpds workspace. No I/O, no database, no config. The crate handles P-256 key generation, `did:key` derivation, AES-256-GCM encryption/decryption, Shamir Secret Sharing (2-of-3 scheme), and [[concepts/did-plc|did:plc]] genesis operation building and verification.

The public API is well-defined with strict format guarantees: `did:key` uses P-256 multicodec varint + multibase base58btc, encryption produces `base64(nonce(12) || ciphertext(32) || tag(16))` = 80 chars, and did:plc genesis op signatures use base64url (no padding) decoding to exactly 64 bytes.

A key design decision is the `build_did_plc_genesis_op_with_external_signer` variant that accepts a signing callback instead of raw private key bytes — enabling signing with non-extractable keys like Apple Secure Enclave.

## Key Takeaways

- **Pure Functional Core**: No I/O, no database, no config. All functions are deterministic given the same inputs (except for random key generation and nonce generation).
- **External signer support**: `build_did_plc_genesis_op_with_external_signer` accepts a `FnOnce(&[u8]) -> Result<Vec<u8>, CryptoError>` callback for non-extractable keys (Secure Enclave).
- **Zeroizing everywhere**: Private key bytes are always `Zeroizing<[u8; 32]>` — callers must not copy into non-zeroizing storage. Shamir shares are also zeroized on drop.
- **Information-theoretic Shamir**: A single share reveals nothing about the secret. GF(2^8) uses AES irreducible polynomial (0x11b).
- **Deterministic DID derivation**: Same inputs → same DID (RFC 6979 ECDSA + SHA-256 + base32). DID format: `did:plc:[a-z2-7]{24}`.
