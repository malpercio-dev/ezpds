# Relay Architecture (CLAUDE.md)

## Summary

Detailed architecture documentation for the [[entities/relay|Relay Server]] crate — the sole Imperative Shell in the workspace. It is the only crate that touches SQLite, handles HTTP, or manages process-level state. All other crates are pure Functional Cores.

The module map defines three layers: `auth/` (authentication primitives, no DB schema ownership), `db/` (SQL query functions by domain entity), and `routes/` (one file per HTTP endpoint, thin Imperative Shell pattern: gather → process → respond).

Hard rules: routes must never import from other routes (shared logic goes to `auth/` or `db/`), every `.rs` file with runtime behavior must have a pattern comment, and `db/` submodules own queries but not transactions (multi-table transactions live in route handlers).

## Key Takeaways

- **Sole Imperative Shell**: The relay is the only crate with I/O boundaries. All other workspace crates are pure Functional Cores.
- **Route isolation**: Routes never import from other routes. Shared logic belongs in `auth/` (pure) or `db/` (queries).
- **Pattern comments mandatory**: Every `.rs` file with runtime behavior must declare `// pattern: Functional Core`, `// pattern: Imperative Shell`, or `// pattern: Mixed (unavoidable)`.
- **DB ownership model**: `db/` submodules own queries, not transactions. Business-logic transactions live in route handlers.
- **Three-layer auth**: `jwt.rs` (Functional Core), `extractors.rs` (Imperative Shell), `dpop.rs` (Mixed). Route handlers call into `auth/`; `auth/` never imports from `routes/`.
